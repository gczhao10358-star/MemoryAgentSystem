# 智忆助理项目面试题库详细版

这份文档按「面试官提问 + 参考回答」形式整理，覆盖项目介绍、架构、检索、画像、提醒、文档处理、异步协作和工程复盘等核心内容。

使用建议：

- 先熟悉“整体项目 + 意图驱动工作流 + 分层记忆架构”三部分，作为主线
- 再重点准备“消息串会话、durable chat 去噪、缓存与持久层一致性”三个工程问题
- 其余问题作为面试深入追问时的扩展回答

## 一、整体项目

### 1. 你先整体介绍一下智忆助理这个项目。

参考回答：

智忆助理 MemoryMate 是一个具备持久记忆和多会话管理能力的 AI 助手系统。它不仅支持普通对话，还能把用户的事实信息、偏好、事件、任务、提醒和文档内容结构化存储下来，并在后续对话中结合当前会话上下文和历史记忆，为用户提供更个性化的回答、日程管理、提醒和文档辅助能力。

### 2. 这个项目的核心目标是什么？解决了什么问题？

参考回答：

这个项目的核心目标，是解决普通大模型“只会当前轮聊天，不真正记住用户”的问题。很多真实使用场景里，用户并不只是想问一个问题，而是希望系统长期理解自己，比如记住自己的偏好、安排、历史事项和沟通风格，所以我围绕“记住什么、怎么存、怎么查、怎么在回答里用”做了一套完整的记忆系统。

### 3. 为什么要做“有记忆能力”的 AI 助手，而不是普通聊天机器人？

参考回答：

普通聊天机器人擅长即时问答，但不擅长长期协作。真正有价值的助手，应该能跨会话持续积累用户信息，比如知道用户最近在做什么项目、偏好什么风格、什么时候有安排、哪些事情要提醒。如果没有记忆能力，系统每轮都像第一次见用户，体验会非常割裂。

### 4. 这个项目里你主要负责哪些模块？

参考回答：

我主要负责整体架构设计和核心链路实现，包括意图驱动工作流、分层记忆架构、内容过滤与记忆分类、混合检索、用户画像学习、多会话隔离、提醒调度，以及会议文档解析流程。后期我还重点修复了多会话消息串线、长期记忆噪声过多、缓存与持久层统计不一致等工程问题。

### 5. 你觉得这个项目最有技术含量的点是什么？

参考回答：

我觉得最有技术含量的点，是把“记忆能力”真正工程化了。不是简单接一个向量库就结束，而是要同时解决记忆入口治理、分层存储、召回融合、用户画像、异步消息、多会话状态一致性等问题。尤其是长期记忆和会话上下文的边界设计，我觉得很能体现系统设计能力。

### 6. 如果让你用一句话概括项目亮点，你会怎么说？

参考回答：

我会说这是一个基于意图驱动工作流、分层记忆架构和混合检索实现的多会话 AI 助手系统，能够把用户长期信息转化为可检索、可演化、可用于回答增强的持久记忆。

## 二、意图驱动工作流

### 1. 你说的“意图驱动工作流”具体是怎么做的？

参考回答：

系统会先对用户输入做意图识别，再根据意图把请求路由到不同工作流。比如 `STORE` 走时间提取、内容润色、冲突检查和结构化存储；`RETRIEVE` 走日期检索和语义检索；`CHAT` 主要走上下文构建和回复生成；`EXECUTE_TASK` 走提醒任务创建。这种方式比所有请求都共用一条链路更清晰，也更容易扩展和调试。

### 2. 这个项目里有哪些主要意图？怎么区分 `store / retrieve / chat / execute_task`？

参考回答：

核心意图有 `STORE`、`RETRIEVE`、`CHAT`、`EXECUTE_TASK`、`DELETE`。区分方式是规则和模型结合：像“提醒我”“查一下我的安排”“删除明天的会议”这种强模式优先规则命中；剩下模糊表达再交给模型去判断意图并抽取关键内容。

### 3. 意图识别是规则做的、模型做的，还是混合做的？

参考回答：

是混合做的。因为纯规则对自然语言变体适应性差，纯模型又不够稳定，所以我采用“规则兜底 + 模型补充”的方式。高置信度场景走规则，复杂语义场景走 LLM，这样能同时兼顾稳定性和理解能力。

### 4. 如果一句话同时包含多个意图，比如“明天提醒我开会，顺便告诉我上周安排”，怎么处理？

参考回答：

这是一个典型的复合意图问题。理想方案是先拆成两个子任务，一个走提醒链路，一个走检索链路。如果当前版本不做完整拆分，我会优先识别主意图，再把剩余信息作为补充处理。后续如果继续优化，我会加一层多意图解析，把一条输入拆成多个 action 并行执行。

### 5. 意图识别错了会带来什么问题？你怎么降低误判？

参考回答：

误判会带来比较直接的业务问题，比如本来只是聊天，却被当成长期记忆写入；或者本来想创建提醒，却没有落到任务调度。所以我尽量让高风险动作优先走规则识别，并在存储前增加冲突检查和确认机制，同时用内容过滤器防止低价值内容误入长期记忆。

### 6. 为什么要做动态路由，而不是所有请求都走同一条链路？

参考回答：

因为不同请求的处理逻辑差异非常大。存储关注结构化和冲突检查，检索关注召回和排序，聊天关注上下文构建，提醒关注时间解析和调度。如果强行塞进一条通用链路，系统会越来越难维护，也不利于定位问题。

## 三、分层记忆架构

### 1. 你为什么设计成“会话记忆 + 近期缓存 + 持久记忆”三层？

参考回答：

因为不同类型的信息生命周期不一样。当前会话上下文只需要保留最近几轮，适合放在会话层；近期高频访问的长期记忆适合放缓存层做性能加速；真正长期有价值的信息才进入持久层。这样拆开以后，系统在性能、成本和信息质量上都更平衡。

### 2. 现在回头看，你觉得一定要三层吗？

参考回答：

不一定非得叫“三层”，但会话层和持久层一定要分开。缓存层本质上是性能优化层，如果项目非常小，前期也可以先做两层；但一旦涉及频繁检索和多次回源，缓存层就会很有价值。所以我现在的看法是，三层不是概念必须，而是工程上比较合理。

### 3. 会话记忆和持久记忆的边界是什么？

参考回答：

会话记忆解决的是“刚刚这段对话里说了什么”，持久记忆解决的是“哪些信息跨会话依然值得记住”。前者偏短期上下文，后者偏长期知识和用户信息，比如偏好、安排、待办和文档内容。

### 4. 什么内容只留在会话里，什么内容进入长期记忆？

参考回答：

普通闲聊、问候、短确认词、即时讨论通常只留在会话里；事实、事件、任务、提醒和文档解析结果这种跨会话仍有价值的信息才进入长期记忆。后来我专门把 durable `chat` 去掉，也是为了强化这个边界。

### 5. 你为什么后来不再把 `chat` 存成长期记忆？

参考回答：

因为我发现普通聊天进入长期记忆会制造大量噪声。真正长期有价值的是结构化信息，而不是所有对话内容。如果什么都存，短期看像“记住很多”，长期看会严重污染检索和回答质量。所以后来我保留了 `CHAT` 作为意图，但不再把它作为 durable memory 类型落库。

### 6. 近期缓存的作用是什么？和持久记忆是不是重复？

参考回答：

缓存层不是业务真相源，它的作用是减少频繁回查数据库和元数据存储的成本，加快热点记忆访问。它和持久记忆保存的是同一类对象，但职责不同：持久层负责可靠存储，缓存层负责访问加速。

### 7. 如果没有缓存层，系统会有什么问题？

参考回答：

没有缓存层并不会让系统不可用，但检索和读取会更频繁地访问 SQLite 和向量库，性能和响应延迟会变差。尤其是搜索结果展示、反复读取热点记忆，或者同一轮流程里多次读同一条记忆时，缓存能明显减少回源次数。

### 8. 这三层之间的数据流转是怎样的？

参考回答：

新消息先进入会话记忆；如果内容被判断为值得长期保留，就走结构化存储进入持久记忆；同时新写入或新读取到的长期记忆会同步进入缓存层。检索时优先拿当前会话上下文，再召回持久记忆，并借助缓存减少回源成本。

### 9. 记忆写入时会经过哪些步骤？

参考回答：

通常会经过内容过滤、意图判断、时间提取、必要的内容润色、冲突检查、结构化分类、持久化存储、缓存同步和画像更新这些步骤。不同类型的记忆会走不同子流程，比如提醒会额外创建调度任务。

### 10. 检索时三层是怎么配合的？

参考回答：

会话层提供当前 session 的最近上下文，持久层提供长期相关记忆，缓存层加速长期记忆读取。最终我会把当前会话、用户画像、相关记忆和必要的置顶信息一起构建成上下文，交给模型生成回复。

## 四、会话记忆

### 1. 会话记忆是怎么实现的？

参考回答：

我是按 `user_id + session_id` 做隔离，每个会话用一个 `deque` 维护最近若干轮消息，形成滑动窗口。这样能保证不同会话之间的上下文不会互相污染。

### 2. 你提到滑动窗口，窗口大小为什么是 20 轮？

参考回答：

20 轮是一个工程上的经验值，能兼顾上下文完整性和 prompt 成本。太小会丢失连续对话信息，太大又会让上下文膨胀。这个值不是绝对的，后续也可以根据模型上下文长度和真实使用情况动态调整。

### 3. “一轮对话”怎么定义？

参考回答：

一般把一次用户输入和对应的助手回复看作一轮，但在实现上会分别记录 user message 和 assistant message，并通过 `turn_id` 把它们关联起来。这样既方便前端渲染，也方便排查异步消息问题。

### 4. 多个 session 之间怎么隔离上下文？

参考回答：

关键是所有会话内状态都必须带 `session_id`。后端记录消息、构建上下文、写提醒结果时都要显式绑定 session；前端渲染消息时也只更新目标 session 对应的消息列表，而不是根据“当前激活会话”去盲目写入。

### 5. 为什么会出现回答串到别的会话？你是怎么修的？

参考回答：

本质原因是异步返回结果和当前前端激活会话混在了一起。修复方式是让请求、响应、typing 状态都显式带 `session_id` 和 `turn_id`，前端只把消息写回对应的 session，不再因为用户切换了界面就把旧请求的结果写到新会话里。

### 6. 用户在模型还没回答完时切换会话，会发生什么？

参考回答：

正确行为应该是原请求继续在后台执行，但返回结果只落到原 session，不应该切走用户当前界面。修完之后，用户切走会话时不会丢消息，也不会自动跳回原会话。

### 7. typing 状态为什么会串会话？你是怎么避免的？

参考回答：

因为最开始 typing 是一个全局状态，而不是 session 级状态。后来我改成按 session 维护 typing 状态，这样某个会话正在生成时，只影响那个会话，不会让别的会话也显示“正在思考”。

### 8. 为什么回答完成后不能自动切回原会话？

参考回答：

因为用户切走会话说明他已经开始关注另一个上下文了，系统不应该强行打断用户当前操作。更合理的做法是把结果落回原 session，必要时给提示，但不改变当前界面焦点。

## 五、近期缓存

### 1. 近期缓存是缓存什么数据？

参考回答：

缓存的是近期写入或读取到的长期记忆条目，也就是完整的 `MemoryEntry`。它不是缓存聊天消息，而是缓存 durable memory，用于减少回查 SQLite 的次数。

### 2. 为什么缓存条目有时候会大于总记忆数？

参考回答：

因为两者一开始统计口径不一致。总记忆数统计的是 SQLite 中当前仍存在的 durable memory，而缓存条目统计的是内存缓存字典长度，可能包含已过期未清理、已删除未同步清除，或者历史 `chat` 条目。后来我统一了统计口径，删除时同步清缓存，统计时也增加了 durable id 交叉校验。

### 3. TTL 和 LRU 是怎么设计的？

参考回答：

缓存会记录 `cached_at` 和访问计数。超过 TTL 会过期清理；如果缓存条目超量，则按访问次数和缓存时间排序，优先淘汰低频且较旧的条目。这个策略结合了时效性和使用热度。

### 4. 缓存何时写入？何时失效？何时淘汰？

参考回答：

新写入长期记忆时会同步写入缓存，读取长期记忆时命中数据库也会回填缓存。缓存条目超过 TTL 会失效；如果总量超过最大容量，就触发淘汰机制移除低优先级条目。

### 5. 缓存命中后和直接查数据库相比，收益在哪里？

参考回答：

最大的收益是减少 SQLite 和元数据存储的回源次数，尤其是在搜索结果展示、重复读取热点记忆、或同一轮流程里多次读同一条记忆时，缓存能明显降低延迟并减少 IO 开销。

### 6. 你怎么保证缓存和持久层一致？

参考回答：

我不把缓存当真相源，而是把持久层当主数据。缓存只做加速，并在删除时同步清理、在统计时做 durable id 校验，尽量避免“缓存里有、数据库里没”的状态影响业务结果。

### 7. 删除持久记忆后，缓存怎么处理？

参考回答：

现在删除 durable memory 后会同步删除缓存中的对应条目，避免页面统计和后续读取出现脏数据。这也是我后面专门修过的一个一致性问题。

## 六、持久记忆

### 1. 持久记忆里有哪些类型？为什么这样划分？

参考回答：

目前主要有 `fact`、`event`、`task`、`reminder`、`document`。这种划分是为了让不同类型的信息在写入、检索和展示时都更清晰，比如事实偏稳定属性，事件偏安排，任务偏执行事项，提醒偏调度动作，文档偏外部内容。

### 2. 什么算 `fact`，什么算 `event`，什么算 `task`，什么算 `reminder`？

参考回答：

`fact` 是用户长期稳定信息，比如偏好、身份、习惯；`event` 是客观安排或要发生的事件；`task` 是需要完成的动作；`reminder` 是显式提醒请求或被调度触发的提醒结果。分类时我主要看时间信息、动作语义和是否包含提醒意图。

### 3. 为什么有时候一句输入会同时提取出多个记忆？

参考回答：

因为一句话里可能同时包含多个信息点，比如既有事件，又有任务，或者会议文档里同时有总结、决策和待办。对于这种复合输入，系统会尽量把长期有价值的信息拆分出来分别落库。

### 4. 为什么“明天有个会议”会被识别成事件而不是事实？

参考回答：

因为它包含明确的时间信息和安排语义，本质上是一个将要发生的日程事件，而不是稳定事实。事实更像“我喜欢喝咖啡”“我是学生”这种长期属性。

### 5. 为什么有些内容不该进入长期记忆？

参考回答：

因为长期记忆不是越多越好，噪声过多会直接拖累检索质量和用户画像准确性。低价值内容进入长期层，后面检索就更容易召回无关信息，所以必须做入口控制。

### 6. 你们是如何避免把低价值内容存进去的？

参考回答：

主要靠内容过滤器、规则分类和 durable chat 下线三层控制。先过滤闲聊和查询，再判断是否属于高价值类型，最终只有符合条件的内容才进入长期记忆。

## 七、内容过滤与记忆分类

### 1. `content_filter` 是怎么工作的？

参考回答：

它本质上是一个启发式内容价值判断模块。先识别闲聊和查询类输入，再通过时间、任务、事实、提醒等规则判断内容更像哪类记忆；如果能命中高价值类别，就允许进入长期层，否则只保留在会话层或者直接过滤掉。

### 2. 你是如何判断一条输入值不值得记忆的？

参考回答：

我主要看三件事：第一，这句话是不是单纯闲聊或查询；第二，它有没有结构化价值，比如偏好、安排、待办、提醒；第三，它是不是具有跨会话复用价值。只有同时满足“不是噪声 + 有结构化价值”，我才会让它进入持久记忆。

### 3. `chat` 为什么设计得不好？后来为什么去掉 durable chat？

参考回答：

因为 `chat` 这个类型太泛了，很多普通聊天、讨论性表达、主观看法都会被它兜住，结果是长期记忆里塞进大量噪声。后来我把 chat 保留成意图，但不再作为 durable 类型，长期层只保留真正有结构化价值的记忆。

### 4. 你怎么区分闲聊、事实、任务、事件？

参考回答：

闲聊通常是问候、确认、寒暄和即时讨论；事实偏稳定属性；事件偏将要发生或已安排事项；任务偏可执行动作。核心判断维度是有没有时间约束、是不是动作事项、是否属于用户长期属性。

### 5. 为什么“我觉得这个方案风险有点大”不应该进长期记忆？

参考回答：

因为它更像当前讨论中的观点表达，而不是稳定事实。除非系统后续要专门支持“观点记忆”或“决策偏好”，否则这类内容进入长期层只会污染检索结果。

### 6. 你怎么处理短确认词，比如“好”“可以”“嗯”？

参考回答：

我把这类内容明确纳入闲聊过滤规则里，避免它们误判成事实或任务。因为这类词语频率高、信息量低，如果进入长期记忆几乎只会制造噪声。

### 7. 规则分类有什么局限？如果继续优化你会怎么做？

参考回答：

局限在于规则对表达变体敏感，也不够语义化。后续我会保留规则做高风险动作兜底，同时增加更稳定的分类模型或可评估的 LLM 分类层，再配合离线样本集做回归测试。

## 八、记忆演化引擎

### 1. 记忆自动衰减与强化是怎么做的？

参考回答：

每条长期记忆都有权重、访问次数、重要性等属性。随着时间推移，如果长期不访问，权重会衰减；如果被反复命中、强化或确认，权重会提升。这样可以让真正有价值的记忆更容易保留下来。

### 2. 什么情况下权重会上升？什么情况下会下降？

参考回答：

被访问、被命中检索、被反复提及或被系统识别为高重要性时会上升；长时间不访问、没有强化、重要性偏低时会下降。这本质上是把“时间因素”和“使用行为”结合起来。

### 3. 这个权重有什么实际作用？

参考回答：

它会影响后续检索排序、展示优先级和记忆保留策略。权重高的记忆更容易进入回答上下文，也更可能在画像和置顶信息里出现。

### 4. 记忆演化是定时跑还是访问时触发？

参考回答：

当前更偏定期执行和访问更新结合。访问时会更新访问计数和访问时间，演化引擎可以按周期批量处理衰减和强化。

### 5. 你怎么避免重要记忆被错误遗忘？

参考回答：

重要记忆不会只看时间，还会结合重要性、访问次数和记忆类型。比如任务、提醒、事实这类本身就有较高业务价值，衰减策略会更保守，不会简单因为一段时间没访问就直接丢掉。

### 6. 如果让你重新设计记忆演化机制，你会怎么改？

参考回答：

我会把“时间衰减、访问强化、用户确认、任务状态”做成更明确的多因子评分模型，并增加不同记忆类型的差异化策略，而不是用相对统一的衰减逻辑。

## 九、混合检索

### 1. 为什么要同时做稠密检索和稀疏检索？

参考回答：

因为二者擅长的场景不同。稠密检索适合找语义相近内容，稀疏检索适合命中精确关键词、日期和专有名词。两者结合后，系统面对自然语言查询和精确查询时都更稳。

### 2. FAISS 和 BM25 分别解决什么问题？

参考回答：

FAISS 负责向量近邻搜索，解决语义相似召回问题；BM25 或关键词检索负责词项匹配，解决精确命中问题。前者更理解语义，后者更擅长关键词和时间表达。

### 3. 两路召回之后怎么融合结果？

参考回答：

两边先分别召回，再按记忆 id 去重，然后综合相似度和业务特征做融合。当前实现更偏简单融合，如果继续优化，我会增加 rerank 层。

### 4. 稠密检索和关键词检索各自的优缺点是什么？

参考回答：

稠密检索优点是语义泛化强，缺点是对精确词和时间短语不稳定；关键词检索优点是可解释、精确，缺点是泛化能力弱。所以单独使用时都会有明显盲区。

### 5. 如果用户问的是精确日期或关键词，哪一路更有优势？

参考回答：

关键词检索更有优势，因为像“4 月 7 日”“周五周报”“飞书提醒”这类查询更依赖精确词项命中。

### 6. 如果用户问的是语义相近但字面不同的问题，哪一路更有优势？

参考回答：

向量检索更有优势，因为它能召回字面不同但语义相近的记忆，比如“我之前说过数据库选型”和“我们讨论过向量库方案吗”。

### 7. 你们如何做 rerank 或去重？

参考回答：

当前至少做了去重，按 memory id 合并不同来源结果。后续如果继续优化，我会在召回后增加基于用户画像、时间相关性和类型权重的 rerank。

### 8. 为什么后来你对 FAISS 有顾虑？

参考回答：

因为 FAISS 更像纯向量索引，不直接承载复杂元数据查询，实际工程里还要依赖 SQLite 回查完整内容和过滤条件，双存储协调成本比较高。

### 9. 如果要换成 Chroma 或 Milvus，你会怎么权衡？

参考回答：

如果是本地原型和轻量部署，我更倾向 Chroma，因为易用性更好；如果是更大规模、分布式或更复杂检索场景，Milvus 会更强，但部署和运维成本也更高。

## 十、向量库与存储选型

### 1. 为什么最初选 FAISS？

参考回答：

因为它轻量、速度快、适合本地原型验证，而且接入成本低。对早期项目来说，先把向量召回能力跑通，比过早引入重型基础设施更重要。

### 2. FAISS 的不足是什么？

参考回答：

不足主要是它对完整业务对象支持不够，核心只负责向量相似搜索，复杂过滤、元数据管理和持久化都要自己补齐，所以工程上会比较割裂。

### 3. 为什么说“只能存向量和 id”会带来麻烦？

参考回答：

因为你最终返回给模型的，不只是一个向量 id，而是完整内容、类型、时间、标签等信息。于是系统就必须再去 SQLite 或元数据层回查，链路会变长，一致性问题也会更多。

### 4. SQLite 在这里承担了什么角色？

参考回答：

SQLite 主要承担元数据存储角色，保存完整记忆内容、类型、时间、权重、会话消息、用户画像和会话信息。它是当前项目里最主要的持久化真相源。

### 5. 如果本地部署，你会选 Chroma 还是 Milvus？为什么？

参考回答：

本地部署我更倾向 Chroma，因为更轻量、更方便，不需要额外上复杂数据库栈。Milvus 更适合规模更大、检索能力要求更高的场景。

### 6. 如果线上化，你会怎么做存储架构拆分？

参考回答：

我会把会话消息、用户画像、业务元数据和向量检索拆开，元数据走关系型数据库，向量索引走专门的向量库，再加缓存层做热点加速。这样职责更清晰，也更容易扩展。

## 十一、多会话与 Session 管理

### 1. session registry 和历史会话列表有什么作用？

参考回答：

它们的作用是管理用户有哪些会话、每个会话的状态是什么、当前该展示哪条历史记录。没有这一层，系统虽然能聊天，但很难支持多线程式对话管理。

### 2. 如何判断是连续会话还是新会话？

参考回答：

当前主要是显式 session 机制，也就是前端切到哪个 session，就把请求发到哪个 session。如果后续要做自动判断，可以结合用户行为、空闲时长和主题变化做会话切分。

### 3. session_id 是怎么生成和管理的？

参考回答：

会在创建新会话时生成唯一 session_id，并在前后端持续透传。前端保存当前激活 session，后端按 `user_id + session_id` 维护会话消息和上下文。

### 4. 为什么新建对话一开始会 404？后来怎么修的？

参考回答：

本质上是前后端在 session 创建或路由上没有完全对齐，导致前端点新建对话后，请求了一个后端还没正确注册的 session。后来我补全了 session 创建接口和状态同步逻辑，保证新会话先建好再切换。

### 5. 会话历史存在哪里？

参考回答：

会话历史最终存储在 SQLite 的 `session_messages` 表里，内存里的会话窗口只是当前运行期加速结构。这样即使服务重启，也能恢复历史消息。

### 6. 为什么退出登录有时不会回到登录页？你是怎么修的？

参考回答：

问题出在前端用户状态、路由跳转和本地缓存没有完全同步清空。后来我把 logout 时的用户信息、当前会话、缓存状态和路由跳转统一收口，避免出现界面还停留在业务页的情况。

## 十二、用户画像

### 1. 用户画像里存了哪些信息？

参考回答：

主要包括话题偏好、专业领域、交互风格偏好、基础行为统计，以及一些用户显式或隐式暴露出来的稳定信息。它既服务于回答个性化，也服务于检索排序。

### 2. 画像里的交互偏好是怎么生效的？

参考回答：

会被转换成系统提示词中的风格约束，比如回答长短、语气正式度、解释深度和主动性。这些信息会在生成回复时一并注入上下文。

### 3. `preferred_response_length`、`preferred_formality`、`preferred_detail_level`、`proactivity_level` 有什么区别？

参考回答：

分别对应“篇幅长短、语气正式程度、技术解释深度、是否主动补充建议”四个维度。它们看起来都影响回答风格，但实际控制的是不同层面的生成行为。

### 4. 这些偏好是硬编码 prompt 规则吗？

参考回答：

可以理解为“规则化的 prompt 注入”，也就是我先定义不同偏好值对应的提示语模板，再根据用户画像动态拼接进去。这样既简单可控，也容易调试。

### 5. 你怎么把用户画像注入到系统提示词里？

参考回答：

我会在构建系统提示词时，把话题偏好、专业领域和交互偏好一起拼接进去，让模型在生成回复时同时参考当前问题、历史记忆和用户风格偏好。

### 6. 画像是怎样从用户对话里学习出来的？

参考回答：

每轮交互后异步提取关键词、更新行为统计、调整话题权重，并根据用户表达更新交互风格字段。这样画像会随着使用逐步演化，而不是一次性静态配置。

### 7. 为什么有些话题偏好会出现脏数据？你是怎么清洗的？

参考回答：

因为如果把 TF-IDF 提取出来的关键词直接当话题，像“提醒、内容、信息、分钟”这种泛词也会进画像。后来我加了词性过滤、规则过滤、标准化和阈值控制，并清洗了 SQLite 里的历史脏数据。

### 8. 话题偏好成为有效话题的阈值是什么？

参考回答：

当前展示层会要求 `weight` 和 `interaction_count` 都达到阈值，只有足够稳定的话题才会真正展示出来。这样能避免偶然词和脏词进入页面。

### 9. 为什么问两次 AI 还不算偏好，三次才进入？

参考回答：

因为系统不是按“出现一次就算偏好”来设计的，而是希望看到重复出现和累计权重，确认这确实是稳定兴趣，而不是偶然提到。

### 10. 专业领域的数据怎么记录？

参考回答：

专业领域更多是从用户长期提问主题、术语分布和交互内容中归纳出来的，最终会作为画像中的 expertise 信息保存，用于检索和回答增强。

## 十三、关键词提取与话题学习

### 1. `text_processor.extract_keywords(...)` 是怎么做的？

参考回答：

当前主要基于 jieba 的 TF-IDF 关键词提取，再配合停用词和词性过滤，把更像主题词的部分保留下来。它是话题学习的第一步，不是最终话题结果。

### 2. 你们用 jieba TF-IDF 提关键词，有没有停用词处理？

参考回答：

有。我后来专门补了停用词和泛词过滤，否则提出来的关键词里会混进很多功能词、时间词和无意义词，最终污染画像。

### 3. `normalize_topic()` 和 `is_meaningful_topic()` 是做什么的？

参考回答：

`normalize_topic()` 用来做标准化，比如去空格、清脏字符、统一格式，避免同一个词被当成多个 topic；`is_meaningful_topic()` 用来判断这个词是否真的适合作为长期话题，比如过滤太短、太泛、纯数字、时间单位和指令词。

### 4. 为什么会提取出很多无意义 topic？

参考回答：

因为关键词提取算法关注的是“统计显著性”，不一定理解“这个词是否值得进入用户画像”。所以关键词不等于话题，必须再做语义和规则层过滤。

### 5. 你怎么清洗历史 topic_preferences 脏数据？

参考回答：

我做了规则过滤脚本，对历史画像数据重新清洗，只保留满足标准化和有效性约束的话题，同时在展示层增加阈值过滤，避免旧脏数据继续暴露给用户。

### 6. 如果让你优化话题学习模块，你会怎么做？

参考回答：

我会增加领域词表、层级归一化、时间衰减、主题聚类和更强的 topic rerank，让话题从“词列表”进一步演化成更稳定的兴趣表示。

## 十四、任务与提醒

### 1. 自然语言时间解析是怎么做的？

参考回答：

系统会优先把“明天、后天下午三点、下周一”这类自然语言时间解析成具体时间对象，再交给提醒调度器。为了减少模型漂移，我后面优先用了本地时间解析器，再让模型做补充。

### 2. 从用户输入到创建提醒，中间有哪些步骤？

参考回答：

一般是意图识别、时间提取、内容润色、冲突检查、必要时请求确认、创建调度任务、落提醒记忆、更新会话消息和画像这几步。

### 3. 如果检测到时间冲突，系统怎么处理？

参考回答：

会先搜索同一时间段已有安排，如果发现冲突就不给直接存，而是先返回确认消息，让用户明确表示是否仍然要添加。

### 4. 请求确认之后，系统怎么知道确认的是哪条待存任务？

参考回答：

我在后端维护了待确认上下文，按 `user_id + session_id` 存储 pending confirmation。用户回复“确认”时，系统会根据这个上下文恢复原始待存内容，而不是重新猜一次。

### 5. 为什么一句话可能生成两条差不多的事件？

参考回答：

通常是因为同一条输入在多个链路里被重复解析或重复写入，比如一次在工作流里落库，另一次又在通用交互存储里落库。后来我专门把 `STORE` 意图下的重复落库问题修掉了。

### 6. 定时提醒消息应该发到哪个会话？为什么不能直接发当前会话？

参考回答：

提醒本质上是异步系统事件，不应该假装成用户当前会话里的同步回复。更合理的做法是按用户维度推送通知，再决定是否把提醒结果写入某个默认或目标 session，而不是直接塞进当前正在看的会话。

### 7. 你后来是怎么处理提醒与会话隔离问题的？

参考回答：

我把提醒和普通会话回复区分开，提醒触发时通过通知通道发送，同时把会话消息落到明确的 session 或默认会话，不再依赖当前前端激活会话状态。

### 8. 异步调度是怎么实现的？

参考回答：

通过后端调度器维护任务，在任务触发时间执行回调，回调里负责消息推送、提醒记忆存储和必要的会话消息记录。这样提醒链路就和普通对话链路解耦了。

### 9. 为什么时间显示有时不是实时变化的？你怎么修的？

参考回答：

因为最开始前端只拿了一次时间，没有持续更新。后来我改成前端定时刷新显示，并结合服务端时间接口对齐，保证界面展示是持续变化的。

## 十五、会议文档处理

### 1. 会议文档处理流水线具体怎么设计的？

参考回答：

流程是先上传和解析文档内容，再做摘要、关键决策和待办提取，最后把结果持久化，并支持把待办转成提醒任务。整个链路里既用到了 LLM，也加了规则提取兜底。

### 2. 为什么要做 LLM 两阶段分析？

参考回答：

因为直接一步抽所有字段不够稳定。两阶段设计可以先做整体理解，再做结构化抽取，提高摘要、决策和待办识别的准确性。

### 3. 第一阶段和第二阶段分别做什么？

参考回答：

第一阶段更偏文档理解和信息整理，比如会议主题、内容总结、关键结论；第二阶段更偏结构化抽取，比如待办事项、负责人、截止时间等。

### 4. 会议摘要、关键决策、待办事项分别怎么提取？

参考回答：

摘要和决策主要走 LLM 分析，待办事项除了 LLM 外，我还补了规则式提取，比如针对列表项、指令式句子、责任人和截止时间模式做识别。

### 5. 为什么有时解析成功了但没有待办事项？

参考回答：

因为并不是每份文档都会用标准格式写待办，LLM 对简短或格式不稳定文档的 action item 抽取也会波动。后来我增加了规则兜底，命中率就提高了。

### 6. 后来你为什么又补了一层规则式待办提取？

参考回答：

因为我发现纯 LLM 提取在一些短会议记录上不稳定，而待办事项是业务上很关键的结果，所以增加规则层可以显著提高 recall。

### 7. 如何判断一个会议文档是否解析过？

参考回答：

通常会根据文档 id、解析结果记录和持久化状态判断，避免重复解析和重复写入同一份文档的内容。

### 8. 为什么会议解析消息一开始没有持久化到会话历史？后来怎么修的？

参考回答：

一开始更多把它当成后台处理结果，而不是正式对话消息，所以只在前端临时展示。后来我发现用户切走会话再回来会看不到，于是把“会议解析完成”消息也写入会话历史表，保证可追溯。

### 9. 会议待办重复的问题是怎么产生的？怎么去重的？

参考回答：

重复通常来自 LLM 结果和规则结果的重叠，或者同一文档多次解析。后来我增加了基于内容相似度和标准化文本的去重逻辑，把重复 action item 合并掉。

## 十六、异步与前后端协作

### 1. 你们哪些地方用了异步？

参考回答：

主要有回复生成、画像更新、提醒调度、消息推送、会议解析以及部分缓存同步。这些步骤里有些在主链路上，有些会放到后台异步任务里执行。

### 2. 为什么要异步处理画像更新、提醒调度、消息推送？

参考回答：

因为这些操作不是当前回答生成的最短关键路径，如果全部同步执行，会拖慢主响应速度。异步处理能让用户更快看到回答，同时后台继续完成画像更新和通知投递。

### 3. 前端为什么会出现消息错乱、切换会话后消息消失？

参考回答：

根因通常是异步响应和会话状态绑定不严，或者前端把消息列表当成全局状态而不是按 session 存。用户切走会话后，旧请求回来如果没有 session 约束，就很容易写错地方。

### 4. 这些前端状态同步问题你是怎么定位和修复的？

参考回答：

我重点看了消息的唯一标识、session 绑定、`turn_id` 和前端 store 的写入逻辑，最后把消息更新改成只作用于目标 session，同时把 typing 也改成 session 级状态，问题就收敛了。

### 5. WebSocket 在项目里承担什么作用？

参考回答：

主要用来做提醒通知和异步消息推送，让后台触发的事件能实时到达前端，而不是只能靠轮询。

### 6. 如果用户离线，消息怎么处理？

参考回答：

当前更合理的方式是走离线消息缓存或第三方通知通道，比如飞书推送。这样用户即使不在当前页面，也能收到提醒。

## 十七、数据库与工程实现

### 1. SQLite 里有哪些核心表？

参考回答：

核心包括记忆表、会话表、会话消息表、用户表和画像相关字段。也就是说，它既承担长期记忆元数据存储，也承担多会话和用户管理的数据存储。

### 2. 初始化时为什么要自动建表？

参考回答：

因为这是一个本地可运行的项目，自动建表可以降低部署和调试门槛，也方便快速验证功能，不需要依赖外部数据库迁移流程。

### 3. 老表和不要的字段为什么可以删？

参考回答：

因为随着架构演化，有些旧设计已经不再使用，比如 durable chat 相关数据和早期兼容字段。如果继续保留，不仅会增加理解成本，也会影响统计和业务口径一致性。

### 4. 为什么重启之后统计一度清零了？后来怎么修的？

参考回答：

因为一开始有些统计依赖运行时内存状态，而不是持久化数据，所以服务重启后内存被清空，统计就归零。后来我改成优先从 SQLite 的 `session_messages` 和 durable memory 恢复统计结果。

### 5. 数据统计里的“近期缓存”和“平均可信度”分别怎么来的？

参考回答：

近期缓存来自内存缓存层中的有效 durable memory 条目数；平均可信度是对当前 durable memory 的 `confidence` 字段做平均计算。两者都不是模型给的单一分数，而是系统维护出来的统计值。

### 6. 为什么缓存条目会比总记忆数还大？你怎么修的？

参考回答：

因为缓存一开始是独立统计内存字典长度，可能包含脏条目，而总记忆数统计的是 SQLite 真实存在的 durable memory。后来我统一了统计口径，删除时同步清缓存，统计时还增加了 durable id 校验。

### 7. 如果以后数据量变大，SQLite 会遇到什么瓶颈？

参考回答：

主要是并发写入能力、复杂查询性能和扩展性会成为瓶颈。到那时我会考虑把会话消息、画像和业务元数据迁到更适合线上场景的数据库，把 SQLite 保留给本地开发或轻量部署版本。

## 十八、系统设计取舍

### 1. 这个项目里你做过哪些架构取舍？

参考回答：

比较典型的取舍包括：规则加 LLM 的混合意图识别、分层记忆而不是单层长期存储、去掉 durable chat、FAISS 加 SQLite 的轻量方案而不是一开始就上复杂向量数据库，以及把很多非关键路径逻辑异步化。

### 2. 为什么不把所有对话都存成长期记忆？

参考回答：

因为长期记忆要强调质量而不是数量。把所有对话都存进去，短期会显得“记得很多”，长期看反而会严重污染检索和回答。

### 3. 为什么不完全依赖 LLM 分类，而保留规则层？

参考回答：

因为像提醒创建、删除安排、时间冲突确认这类场景都属于高风险动作，纯 LLM 不够稳定。规则层可以提供确定性和可解释性，模型层负责复杂语义理解。

### 4. 为什么要把记忆系统拆成 session / cache / durable 三层，而不是两层？

参考回答：

因为会话上下文和长期知识本身就不是同一种信息，缓存又是典型的性能层。三层拆开后职责更清楚，调优时也更好定位问题。

### 5. 如果让你重新设计一版，你会删掉什么、保留什么？

参考回答：

我会保留意图驱动工作流、会话隔离和 durable memory 边界设计；会弱化早期一些兼容命名和多余类型，比如 durable chat；存储层可能会换成更一体化的元数据加向量方案。

### 6. 你觉得这个项目现在最大的技术债是什么？

参考回答：

我觉得最大的技术债是存储层仍然有一定双轨制复杂度，也就是向量索引和业务元数据要协调；另外一些分类和画像学习逻辑还偏启发式，后续需要更系统的评估和数据闭环。

## 十九、难点与复盘

### 1. 项目里你遇到的最难问题是什么？

参考回答：

我觉得最难的是多会话异步消息一致性问题。因为它跨越前端状态管理、后端异步返回和会话隔离三个层面，单看某一层都不容易发现根因。

### 2. 你是怎么定位“消息串会话”这个 bug 的？

参考回答：

我先复现出“在会话 1 发送消息、切到会话 2、回答却跑到当前界面”的路径，然后沿着 `session_id`、`turn_id`、typing 状态和前端消息写入逻辑逐层排查，最后定位到问题是返回结果没有严格绑定目标 session。

### 3. 你做过哪些“看起来不是大功能，但很能体现工程能力”的修复？

参考回答：

比如 durable chat 下线、缓存统计口径统一、删除时同步清缓存、会议解析结果持久化到会话历史、前端自动重建 dist、防止 typing 串会话，这些都不是大功能，但很体现系统性和工程细节能力。

### 4. 如果面试官让你讲一个印象最深的 bug，你讲哪个？

参考回答：

我会讲多会话消息串线。因为它很典型地体现了异步系统里“请求发起时上下文”和“响应落地时上下文”不一致的问题，而且修复过程也能展示我对前后端状态一致性的理解。

### 5. 这个项目上线后你最想继续优化的 3 个点是什么？

参考回答：

第一，升级存储层，把向量索引和元数据管理整合得更自然；第二，给意图识别、记忆分类和画像学习建立评估集；第三，进一步完善提醒通知和多端同步机制，让它更像真正可长期使用的个人助手。

# 智忆助理项目面试题库核心问题版

这份文档保留“快速过一遍”的特点，但不只列问题，而是给每个问题补一行回答关键词。

适合场景：

- 面试前 10 到 20 分钟快速复习
- 先看提纲，再口头复述
- 忘了怎么答时，快速找回表达主线

使用方法：

1. 先看问题，自己尝试口头回答
2. 再看下面的“回答关键词”
3. 如果还想展开，再去看详细版

---

## 一、必背主线 15 题

### 1. 你先介绍一下这个项目。

回答关键词：

- 持久记忆 + 多会话管理
- 事实 / 事件 / 任务 / 提醒 / 文档结构化存储
- 结合当前会话和历史记忆做个性化回答

### 2. 你为什么要做这个项目？

回答关键词：

- 解决普通大模型只会当前轮聊天的问题
- 希望 AI 能长期理解用户
- 从即时问答工具升级成长期协作助手

### 3. 这个项目最大的亮点是什么？

回答关键词：

- 意图驱动工作流
- 分层记忆架构
- 混合检索
- 多会话隔离与异步消息

### 4. 你说的意图驱动工作流具体是什么？

回答关键词：

- 先识别意图，再动态路由
- `STORE / RETRIEVE / CHAT / EXECUTE_TASK / DELETE`
- 不同请求走不同链路，逻辑更清晰

### 5. 为什么不用纯大模型，而要做规则加模型的混合路由？

回答关键词：

- 纯模型不稳定
- 动作型任务风险高
- 规则负责确定性，模型负责复杂自然语言理解

### 6. 你为什么设计成会话记忆、近期缓存、持久记忆三层？

回答关键词：

- 信息生命周期不同
- 会话层保短期上下文
- 缓存层做性能优化
- 持久层保长期有价值信息

### 7. 现在回头看，一定要三层吗？

回答关键词：

- 会话层和持久层一定要分开
- 缓存层本质是性能层
- 小项目可先两层，大一点三层更合理

### 8. 你为什么后来不再把 `chat` 存成长期记忆？

回答关键词：

- 普通聊天噪声太多
- 会污染检索和画像
- `CHAT` 保留为意图，不再作为 durable type

### 9. 会话记忆是怎么实现的？

回答关键词：

- `user_id + session_id` 隔离
- 每个 session 一个 `deque`
- 维护最近若干轮消息

### 10. 为什么会出现消息串会话？你是怎么修的？

回答关键词：

- 异步返回结果没有严格绑定 session
- 消息和 typing 都加 `session_id + turn_id`
- 前端只更新目标会话，不再写当前激活会话

### 11. 为什么要做混合检索？

回答关键词：

- 向量检索语义强，精确词不稳
- 关键词检索精确强，语义泛化差
- 两路结合提高召回率和鲁棒性

### 12. 你们的混合检索具体怎么做？

回答关键词：

- 稠密检索：FAISS 1024 维
- 稀疏检索：BM25 / 关键词匹配
- 双路召回后去重融合

### 13. 用户画像是怎么做的？

回答关键词：

- 话题偏好、专业领域、交互风格、行为统计
- 每轮交互后异步更新
- 用于提示词增强和检索排序

### 14. 自然语言提醒和异步调度是怎么做的？

回答关键词：

- 先做时间解析
- 再创建调度任务
- 到点后异步回调
- WebSocket / 飞书推送提醒

### 15. 如果现在再做一版，你最想优化什么？

回答关键词：

- 向量库和元数据进一步整合
- 意图识别和记忆分类可评估化
- 提醒、通知、会话消息进一步事件总线解耦

## 二、高频追问 20 题

### 16. 这个项目里有哪些主要意图？怎么判断？

回答关键词：

- `STORE / RETRIEVE / CHAT / EXECUTE_TASK / DELETE`
- 强模式规则命中
- 复杂表达交给 LLM

### 17. 如果一句话同时包含多个意图，比如“提醒我开会，顺便查上周安排”，怎么处理？

回答关键词：

- 属于复合意图
- 理想方案是拆子任务
- 当前可先识别主意图，再补充次意图

### 18. 会话记忆和持久记忆的边界是什么？

回答关键词：

- 会话层：刚刚这段对话说了什么
- 持久层：跨会话依然有价值的信息
- 核心是“短期连续性”和“长期复用性”的区别

### 19. 什么内容会进入长期记忆？什么不会？

回答关键词：

- 会进：`fact / event / task / reminder / document`
- 不会进：闲聊、短确认词、普通问候、纯查询

### 20. 你们怎么判断一条内容值不值得记忆？

回答关键词：

- `content_filter`
- 先过滤闲聊和查询
- 再判断是不是高价值类型

### 21. 你怎么区分 `fact`、`event`、`task`、`reminder`？

回答关键词：

- 看时间约束
- 看是不是动作事项
- 看是不是稳定属性
- 看有没有显式提醒语义

### 22. 近期缓存是缓存什么数据？

回答关键词：

- 缓存的是近期访问或写入的 durable memory
- 不是聊天消息缓存
- 作用是减少回查 SQLite

### 23. 为什么缓存条目有时会大于总记忆数？

回答关键词：

- 统计口径不一致
- 缓存里可能有过期或已删除未清理条目
- 后来统一了统计口径并同步清缓存

### 24. TTL 和 LRU 是怎么设计的？

回答关键词：

- TTL 控制时效性
- 超量时按访问次数和时间淘汰
- 兼顾热度和新鲜度

### 25. 为什么最开始选 FAISS？后来又觉得它不够好？

回答关键词：

- 前期轻量、快、适合原型
- 后期发现只擅长向量索引
- 完整内容和过滤还得依赖 SQLite

### 26. SQLite 在这个项目里承担什么角色？

回答关键词：

- 元数据真相源
- 存记忆内容、画像、会话、消息
- 向量库只负责相似检索，不负责完整业务对象

### 27. 交互偏好里的四个维度分别控制什么？

回答关键词：

- 回答长度
- 语气正式度
- 解释深度
- 主动补充程度

### 28. 这些偏好是怎么真正作用到回答生成上的？

回答关键词：

- 动态拼接到系统提示词
- 不同值映射成不同风格约束
- 真正接入生成链路

### 29. 话题偏好是怎么学习出来的？

回答关键词：

- 提关键词
- 做归一化和有效性过滤
- 更新权重和交互次数
- 达到阈值才展示

### 30. 为什么问两次 AI 还没进入话题，三次才进？

回答关键词：

- 不是一次出现就算偏好
- 要满足 `interaction_count` 和 `weight` 阈值
- 目的是防止噪声进入画像

### 31. 会议文档处理流水线是怎么设计的？

回答关键词：

- 先解析文档
- LLM 做摘要、决策和待办抽取
- 再补规则式待办提取兜底

### 32. 为什么有时会议解析成功但没有待办事项？

回答关键词：

- 纯 LLM 对格式敏感
- 短文档和非标准纪要不稳定
- 后来加规则识别和去重

### 33. 为什么提醒消息不应该直接发到用户当前会话？

回答关键词：

- 提醒是异步系统事件
- 直接塞进当前会话容易串线
- 更合理的是按用户通道推送

### 34. 你们哪些地方用了异步？

回答关键词：

- 回复生成
- 画像更新
- 提醒调度
- 消息推送
- 会议解析

### 35. 为什么要异步处理画像更新、提醒调度、消息推送？

回答关键词：

- 不阻塞主回答链路
- 降低响应延迟
- 后台继续做非关键路径工作

### 36. 前端为什么会出现消息错乱、切换会话后消息消失？

回答关键词：

- 会话状态绑定不严
- 异步响应和当前激活会话混在一起
- 消息列表一开始更像全局状态

### 37. 这些前端状态同步问题你是怎么定位和修复的？

回答关键词：

- 看 `session_id / turn_id / typing`
- 排查 store 写入逻辑
- 改成消息只落目标会话

### 38. 为什么重启之后统计一度清零了？后来怎么修的？

回答关键词：

- 一开始依赖运行时内存
- 重启后内存清空
- 后来改成优先从 SQLite 恢复统计

### 39. 这个项目里你做过哪些架构取舍？

回答关键词：

- 规则 + LLM 混合意图识别
- 分层记忆而不是单层存储
- 去掉 durable chat
- FAISS + SQLite 轻量方案

### 40. 你在这个项目里遇到的最有代表性的工程问题是什么？

回答关键词：

- 多会话异步消息串线
- durable chat 噪声
- 缓存与持久层一致性

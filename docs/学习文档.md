# 学习文档

## 1. 现在的话题是怎么学习出来的

用户问题
-> jieba TF-IDF 提关键词
-> 去停用词
-> 再按词性过滤
-> 再按话题规则过滤
-> 才进入 topic_preferences

### 1.1 候选词提取

文件：

- `memory_assistant/profile/profile_learner.py`

核心做法：

1. 先用 `text_processor.extract_keywords()` 提取关键词
2. 再用 `jieba.posseg` 对原句做词性标注
3. 只优先保留更像主题的词性

目前保留的词性主要有：

- `n` 普通名词
- `nr` 人名
- `ns` 地名
- `nt` 机构名
- `nz` 专有名词
- `vn` 名动词
- `eng` 英文技术词

这样能保留：

- 数据库
- 飞书
- Milvus
- 用户画像
- 向量数据库

同时会过滤掉很多动作词和功能词。

### 1.2 有意义话题过滤

文件：

- `memory_assistant/utils/topic_utils.py`

这里增加了 `is_meaningful_topic()` 和 `sanitize_topic_preferences()`。

主要规则包括：

- 长度必须在合理区间内
  - 太短不行
  - 太长也不行
- 过滤停用词和泛词
  - 如：`提醒`、`分钟`、`信息`、`内容`、`结果`、`详情`
- 过滤时间和计量单位
  - 如：`5分钟`、`3次`、`2个`
- 过滤纯数字或纯符号
- 过滤一些指令式前缀
  - 如：`帮我xx`、`请帮xx`

### 1.3 写入画像前再次拦截

文件：

- `memory_assistant/models/user_profile.py`

`add_or_update_topic()` 现在不再直接写入，而是：

1. 先 `normalize_topic()`：先把候选话题做格式标准化，避免同一个词因为空格、符号、脏字符不同，被当成不同话题。
2. 再 `is_meaningful_topic()`
3. 不满足规则就直接丢弃
4. 满足规则才写入 `topic_preferences`
5. 写完后再执行一次 `sanitize_topic_preferences()`

这一步的意义是：

- 即使上游漏掉了脏词，也不会真正进入画像

### 1.4 展示层阈值过滤

文件：

- `memory_assistant/profile/profile_manager.py`
- `memory_assistant/utils/topic_utils.py`

页面展示前，不是直接把所有画像话题都返回，而是再做一次过滤：

- `weight >= 0.6`
- `interaction_count >= 2`

也就是说，某个词必须：

- 足够重要
- 出现不止一次

才会在页面里真正显示。

## 2. 哪些词现在更可能成为“话题”

更可能成为话题的词：

- 数据库
- 数据库优化
- 飞书机器人
- 向量数据库
- 会议纪要
- RAG
- Chroma
- Milvus
- 用户画像
- 前端交互

不太可能成为话题的词：

- 提醒
- 设置
- 查看
- 帮我
- 信息
- 内容
- 分钟
- 一下
- 这个
- 那个

## 3 这次问题的核心经验

### 3.1 关键词不等于话题

能被 TF-IDF 或关键词算法提取出来的词，不一定适合作为用户画像中的长期兴趣。

### 3.2 画像数据必须做入口治理

用户画像一旦被脏数据污染，会影响：

- 前端展示
- 检索偏好
- Prompt 个性化
- 用户对系统可信度的判断

所以不能只在展示层“遮一遮”，必须在写入前就做过滤。

### 3.3 展示层还要再做一层阈值控制

画像内部可以保留候选话题，但真正对用户展示时，要只展示更稳定、更有代表性的内容。

### 3.4 数据库清理不等于线上状态清理

如果服务端有内存缓存，数据库已经变更，并不代表接口返回一定已经更新。

## 4. 面试时可以怎么讲

可以这样回答：

> 我在做用户画像时，发现“话题偏好”被很多无意义词污染，比如“提醒、分钟、信息”这种。根因是系统把关键词提取结果直接当成话题，没有区分主题词和功能词。我后来做了三层治理：第一层在学习阶段加入词性过滤，只保留更像名词和专有词的候选词；第二层在写入画像前做规则过滤，屏蔽停用词、时间单位、泛词和指令词；第三层在展示阶段增加阈值控制，只有权重和交互次数都达标的话题才显示。同时我写了脚本清理 SQLite 里的历史脏画像，并处理了后端内存缓存导致的旧数据回显问题。这样最终把画像从“关键词列表”改造成了更稳定的“兴趣话题表示”。

## 5. 后续还能继续优化什么

下一步可以继续做：

- 增加领域白名单
  - 技术、产品、日程、健康、学习等
- 增加时间衰减
  - 老话题长期不出现，自动降权
- 增加“话题归一化”
  - 例如：`数据库`、`MySQL`、`SQL 优化` 建立层级关系
- 增加可观测性
  - 提供一个调试接口，输入一句话就能看到会学出哪些候选话题

## 6. 一句话总结

这次不是简单“删掉几个脏词”，而是把“关键词提取 -> 画像写入 -> 页面展示 -> 历史数据清洗 -> 缓存一致性”整条链路一起改成了更可靠的画像学习流程。
